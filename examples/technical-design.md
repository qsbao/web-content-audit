# 用户权限系统重构技术方案

## 背景与目标

当前权限系统基于硬编码角色（admin/user/guest），无法满足多租户场景下的细粒度权限控制需求。

**目标**：
- 支持RBAC（基于角色的访问控制）+ ABAC（基于属性的访问控制）混合模型
- 支持多租户隔离
- 权限变更实时生效，无需重启服务

## 现状分析

### 现有架构

```
用户 → 硬编码角色(admin/user/guest) → 路由中间件鉴权
```

### 存在的问题

1. 角色粒度太粗，无法区分"只读财务数据"和"可编辑财务数据"
2. 新增角色需要改代码发版
3. 不支持租户间权限隔离，A公司的admin可以看到B公司的数据

## 方案设计

### 数据模型

```
Tenant(租户) 1:N Role(角色) N:N Permission(权限)
User(用户) N:N Role(角色)
Permission = resource(资源) + action(操作) + condition(条件)
```

### 核心表结构

| 表名 | 字段 | 说明 |
|------|------|------|
| permissions | id, resource, action, condition | 权限定义 |
| roles | id, tenant_id, name, description | 角色定义，租户隔离 |
| role_permissions | role_id, permission_id | 角色-权限关联 |
| user_roles | user_id, role_id | 用户-角色关联 |

### 鉴权流程

1. 请求到达API网关
2. 解析JWT获取user_id和tenant_id
3. 从Redis缓存查询用户权限集合（缓存未命中则查DB）
4. 匹配请求的resource+action是否在权限集合中
5. 若有condition，进一步评估ABAC条件（如：只允许查看本部门数据）

## 影响范围

- API网关鉴权中间件：需要重写
- 用户管理后台：新增角色/权限管理页面
- 现有接口：需要标注resource+action元数据（约120个接口）

## 里程碑

| 阶段 | 内容 | 时间 |
|------|------|------|
| P1 | 数据模型 + 鉴权中间件 | W13-W14 |
| P2 | 管理后台 + 现有接口标注 | W15-W16 |
| P3 | ABAC条件引擎 | W17 |
| P4 | 灰度上线 + 旧系统兼容 | W18 |

## 风险

1. **接口标注工作量大** — 120个接口需要逐一标注，建议按模块分配给各团队
2. **缓存一致性** — 权限变更后Redis缓存需要及时失效，计划使用发布订阅机制
3. **兼容期维护成本** — 新旧系统并行期间需要同时维护两套鉴权逻辑
